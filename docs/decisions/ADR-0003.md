**Status:** accepted
**Deciders:** Matt Watt

## Context

toke programs need to serialize and deserialize structured data for I/O, LLM
context windows, configuration, and user-facing text. JSON is the dominant
format but wastes 40--70% of available context window tokens on formatting
overhead (repeated keys, braces, quotes, brackets). As toke is designed for
LLM code generation, token efficiency in data payloads is as important as
token efficiency in source code.

Research into JSON alternatives (TOON, TRON, YAML, CSV/TSV, Markdown) shows
that no single format universally beats compact JSON. Instead, optimal format
selection depends on data shape: tabular data benefits from TOON (30--60%
savings), nested config data from YAML (18--30%), and heterogeneous data from
minified JSON.

Additionally, toke's design principle that code generators should focus on
code -- not strings -- requires a mechanism for externalising user-facing text
and supporting internationalisation without embedding string literals in
source code.

## Decision

1. **Three-tier format hierarchy:** TOON is the default serialization format.
   YAML and JSON are secondary formats. All three are first-class stdlib
   modules (`std.toon`, `std.yaml`, `std.json`) with consistent interfaces.

2. **TOON as default:** For the tabular data patterns common in LLM contexts
   (database results, API response arrays, agent state, RAG pipelines), TOON
   achieves 30--60% fewer tokens than JSON by declaring field names once in a
   schema header and using pipe-delimited value rows.

3. **Common module interface:** All serialization modules follow the same
   function pattern: `enc`, `dec`, typed accessors (`str`, `i64`, `f64`,
   `bool`, `arr`), and cross-format conversion (`from_json`, `to_json`).
   This pattern is extensible -- additional formats (TRON, CSV, Markdown) can
   be added as plug-in modules following the same interface.

4. **String externalisation via std.i18n:** User-facing text is stored in
   external data files (TOON/YAML/JSON bundles) rather than embedded in toke
   source. The `std.i18n` module loads locale-specific bundles with automatic
   fallback and provides `{placeholder}` substitution. Internationalisation
   is a core capability, not an add-on.

5. **Format-agnostic bundle loading:** The i18n module auto-detects format
   from file extension, searching TOON first, then YAML, then JSON. This
   allows teams to choose the bundle format that best fits their workflow
   without changing application code.

## Rationale

- **Token efficiency compounds at scale.** A RAG pipeline processing thousands
  of records per request burns significant cost on JSON formatting overhead.
  TOON's tabular layout eliminates this for the most common data shape.

- **No single format wins universally.** Independent benchmarks show TOON
  excels for uniform arrays of objects but can lose to minified JSON on deeply
  nested, heterogeneous data. Supporting multiple formats with data-shape-
  dependent selection avoids locking into a suboptimal choice.

- **Code generators should not generate strings.** Existing LLM models and
  tokenizers already handle natural language text well. Toke code stays
  lightweight by externalising variable text, letting existing models handle
  string content while toke handles logic.

- **Consistent interfaces reduce cognitive load.** A developer who knows
  `json.str(j, "key")` immediately knows `toon.str(t, "key")` and
  `yaml.str(y, "key")`. The common interface also makes format migration
  trivial.

## Consequences

**Positive:**
- Programs can switch serialization formats by changing one import line
- Token costs for structured data in LLM contexts reduced by 30--60%
- Internationalisation works out of the box without additional dependencies
- The extensible interface allows community-contributed format modules

**Negative:**
- Three format implementations to maintain in the stdlib
- TOON is not yet standardised (described as "an idea in progress" as of 2026)
- LLMs trained primarily on JSON may need few-shot examples for TOON I/O
- Bundle file discovery adds filesystem I/O at startup

## Alternatives considered

1. **JSON only:** Simplest, but ignores 30--60% token savings for tabular data.
   Rejected because token efficiency is a core project goal.

2. **TOON only:** Maximum token savings, but poor fit for nested configuration
   data and breaks interop with JSON-expecting APIs. Rejected in favour of
   a multi-format approach.

3. **YAML only:** Good readability but actually costs 6--10% more tokens than
   minified JSON. Rejected as sole format; retained as secondary for config
   data where readability matters.

4. **Embedded string literals for i18n:** Standard approach in most languages,
   but contradicts toke's principle of keeping source code focused on logic.
   Rejected in favour of externalised bundles.
