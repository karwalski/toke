# ADR-0002: String Interpolation — Deferred to Profile 2

**Date:** 2026-03-28
**Status:** accepted
**Deciders:** M. Karwalski

## Context

The toke spec describes string interpolation syntax `\(expr)` in string literals. Three implementation options exist:
- Option A: Lexer mode stack (emit STR_START/Expr/STR_END tokens)
- Option B: Post-parse expansion pass
- Option C: Defer to Profile 2

## Decision

Option C — defer string interpolation to Profile 2. Emit W1010 warning if `\(` appears in a Profile 1 string literal.

## Rationale

Option A is correct long-term but adds a mode stack and parenthesis depth counter to the lexer, increasing complexity at the moment when the lexer must be simple for corpus generation. Option B damages diagnostic position accuracy — embedded expressions produce source positions offset from the original file, breaking the `pos`, `line`, `column` fields in structured diagnostics. Accurate diagnostic positions are load-bearing for the automated repair loop.

The deferral is viable because the corpus generation pipeline controls LLM output. The system prompt will instruct models to use `str.concat()` instead of interpolation. W1010 provides a clean diagnostic if a generated program uses interpolation anyway.

## Profile 2 implementation plan

Use Option A: the lexer enters an interpolation mode on `\(`, emits `STR_START`, then lexes the nested expression with a parenthesis depth counter, emits `STR_END` on the matching `)`, and continues with the remainder of the string. Nested interpolation is prohibited (depth > 1 emits E1011).

## Consequences

- Profile 1 lexer stays under 300 lines with no mode stack
- Diagnostic positions are always accurate in Profile 1
- LLM-generated programs using interpolation fail with a clear, actionable warning
- Profile 2 lexer will require a mode stack — plan for this in the Profile 2 lexer story
