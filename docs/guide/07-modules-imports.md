## Module declarations revisited

Every `.tk` file starts with a module declaration:

```
m=app.user;
```

The module path serves two purposes:

1. **Identity** -- it names the module globally within the project
2. **Resolution** -- it maps to the file system (by convention, `app.user` lives at `app/user.tk`)

Module paths are dot-separated lowercase identifiers. Keep them short and descriptive.

## Imports

The `i=` declaration imports a module and gives it a local alias:

```text
i=http:std.http;
i=db:std.db;
i=user:app.user;
```

After importing, you access the module's exports through the alias:

```
let req=http.$req{method:"GET";path:"/users"};
let u=user.getbyid(42);
```

### Import rules

- All imports must come after the module declaration and before any type, constant, or function declarations
- There are no wildcard imports -- every import has an explicit alias
- The alias is a local name; it does not affect other files
- Circular imports are a compile error (E2008)

## Standard library modules

The standard library is accessed through the `std.*` namespace:

| Module | Purpose |
|--------|---------|
| `std.str` | String operations (len, slice, split, join, replace, etc.) |
| `std.io` | Console I/O (println, print, readline) |
| `std.file` | File system operations (read, write, append, exists) |
| `std.json` | JSON encoding and decoding |
| `std.http` | HTTP server and client |
| `std.db` | Database access (SQLite, query, execute) |
| `std.net` | TCP socket operations |
| `std.crypto` | Hashing and cryptographic utilities |
| `std.process` | Run external commands and manage processes |
| `std.time` | Timestamps, durations, and formatting |
| `std.math` | Mathematical functions (sqrt, abs, pow, etc.) |

Import only what you use. Each import adds to the compilation context, so keeping imports minimal helps both the compiler and LLMs working with the file.

## Interface files (.tki)

When you import a module, the compiler does not need the full source code of that module -- it only needs the **interface file**. Interface files have the `.tki` extension and contain only the public type and function signatures:

Generated by the compiler as `user.tki`:

```
m=app.user;

t=$user{id:u64;name:$str;email:$str};
t=$usererr{$notfound:u64;$dberror:$str};

f=getbyid(id:u64):$user!$usererr;
f=create(name:$str;email:$str):$user!$usererr;
f=del(id:u64):void!$usererr;
```

Notice that function declarations in `.tki` files have no body -- just the signature followed by `;`.

### How interface resolution works

When the compiler encounters `i=user:app.user;`, it:

1. Looks for `app/user.tki` in the interface search path
2. Parses the type and function signatures
3. Makes them available under the `user` alias
4. Does not need `app/user.tk` to type-check the current file

This is what makes toke **incrementally compilable** -- each file can be checked independently given only its import interfaces. This property is critical for LLM workflows where you want to generate and validate one file at a time.

### Generating interface files

The compiler generates `.tki` files automatically during compilation:

```bash
tkc app/user.tk    # compiles user.tk and produces app/user.tki
```

In a multi-file project, you compile each file and the interfaces are produced as a side effect.

## Multi-file project structure

Here is a typical toke project layout:

```
myapp/
  main.tk           m=myapp.main
  user.tk           m=myapp.user
  db.tk             m=myapp.db
  config.tk         m=myapp.config
```

The main entry point:

`main.tk`:

```
m=myapp.main;
i=io:std.io;
i=user:myapp.user;
i=config:myapp.config;

f=main():i64{
  let cfg=mt config.load("config.json") {
    $ok:c   c;
    $err:e  {
      io.println("Failed to load config");
      <1;
    };
  };
  io.println("App started on port \(cfg.port as $str)");
  <0;
};
```

The user module:

`user.tk`:

```
m=myapp.user;
i=db:myapp.db;

t=$user{id:u64;name:$str;email:$str};
t=$usererr{$notfound:u64;$dberror:$str};

f=getbyid(id:u64):$user!$usererr{
  let row=db.queryone("SELECT * FROM users WHERE id=?";@(id))!$usererr;
  <$user{id:row.u64("id");name:row.str("name");email:row.str("email")};
};

f=create(name:$str;email:$str):$user!$usererr{
  let row=db.exec("INSERT INTO users (name,email) VALUES (?,?)";@(name;email))!$usererr;
  <$user{id:row.lastid;name:name;email:email};
};
```

The config module:

`config.tk`:

```
m=myapp.config;
i=file:std.file;
i=json:std.json;

t=$config{host:$str;port:i64;debug:bool};
t=$configerr{$fileerr:$str;$parseerr:$str};

f=load(path:$str):$config!$configerr{
  let content=file.read(path)!$configerr;
  let cfg=json.dec(content)!$configerr;
  <cfg;
};
```

### The one-export-per-file convention

toke strongly encourages each file to export one primary construct -- one main type or one main function. This is a linter convention (L0001), not a compiler requirement.

Why? It keeps each file small enough for an LLM to generate in a single pass. The generation context for one file is bounded to:
- The interface types of its imports (~40 tokens each)
- The function signature being implemented (~20 tokens)
- The task description

A typical generation context is under 500 LLM tokens.

## Building multi-file projects

Compile all files together:

```bash
tkc main.tk user.tk db.tk config.tk -o myapp
```

Or compile incrementally -- each file produces its `.tki`, and dependent files use those interfaces:

```bash
tkc config.tk          # produces config.tki
tkc db.tk              # produces db.tki
tkc user.tk            # uses db.tki, produces user.tki
tkc main.tk            # uses user.tki, config.tki
tkc --link -o myapp    # links all compiled objects
```

### Exercise 2: Write an interface

Without looking at the compiler output, write the `.tki` interface file for this module by hand:

```
m=math.vec;

t=$vec2{x:f64;y:f64};

f=add(a:$vec2;b:$vec2):$vec2{
  <$vec2{x:a.x+b.x;y:a.y+b.y};
};

f=dot(a:$vec2;b:$vec2):f64{
  <a.x*b.x+a.y*b.y;
};

f=scale(v:$vec2;s:f64):$vec2{
  <$vec2{x:v.x*s;y:v.y*s};
};
```

### Exercise 3: Three-module project

Design a project with three modules:
- `todo.model` -- defines `t=$task{id:u64;title:$str;done:bool}`
- `todo.store` -- provides `add`, `complete`, and `list` functions using an in-memory array
- `todo.main` -- wires it together with a simple command loop

Write the module declarations and import statements for each file. You can stub the function bodies.

## Key takeaways

- `i=alias:module.path;` imports a module under a local alias
- Standard library lives under `std.*` (11 modules)
- Interface files (`.tki`) contain only public signatures -- no bodies
- Each file is independently type-checkable given its import interfaces
- One primary export per file is the recommended convention
- Module paths map to file system paths by convention
- No wildcard imports, no circular imports
